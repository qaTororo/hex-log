# 要件定義書 (Requirements)

## 1. プロジェクト概要

- **プロジェクト名:** Hex Log
- **目的 (Why):**
  1. **ポートフォリオ:** 将来の転職活動時などに、採用担当者に技術力（設計・実装・テスト・運用）を示すための成果物とする。
  2. **ナレッジ蓄積:** 自身の学習記録（TFT、技術）を継続的に残す。インプレッション数は追わず、宣伝もしない。
  3. **実験場:** 新しい技術（Hono, Bun, React, Tailwind CSS, Figma AIなど）を試すサンドボックスとする。

## 2. ターゲットユーザー (Who)

- **プライマリ:** 面接官・採用担当者（技術的な品質やこだわりを見る人）
- **セカンダリ:** 未来の自分（過去の思考やログを振り返る人）
- **対象外:** 一般のTFTプレイヤー（攻略情報としての価値は主目的としない）

## 3. デザインコンセプト (What)

- **キーワード:** Personal, Experimental, Hextech
- **実装方針:**
  - **Tailwind CSS** を採用し、ゼロからスタイリングを行う。
  - デザイン作成には **Figma (AI / MCP)** を活用し、モダンなデザインフローを実践する。
  - うまくいかない場合のフォールバックとして Chakra UI を検討する。

## 4. 機能要件 (How)

### Must (MVP: Minimum Viable Product)

- **記事管理:**
  - Markdownファイルベースの管理。
  - Gitによるバージョン管理（ブログシステム自体と一緒に管理）。
- **記事閲覧:**
  - トップページ（記事一覧、日付順）。
  - 記事詳細ページ（Markdown → HTML変換）。
- **メタデータ:**
  - OGP設定（チャットツール等でリンクを送った際にリッチに表示されること）。
- **品質保証:**
  - CI/CDによる自動テスト（Bun test）。

### Should / Could (今後の拡張)

- **TFT特化機能:**
  - 専用コンポーネント（盤面配置、アイテムアイコン等）はMVPには含めず、ブログ自体のバージョンアップとして順次実装する。
- **検索・フィルタリング:**
  - 記事数が増えてきた段階で実装。
